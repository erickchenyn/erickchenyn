最近随着 moxt 的不断迭代，我关注了更多来自用户的真实反馈，证实了一个其实我早就预料到的问题。

moxt 的叙事之一是“为 AI 提供友好的 context”，这使得 moxt 坚持走 md、csv、html（对标 word、excel、ppt）的路子。但绝大多数这个阶段上手 moxt 的人 —— 注意是人，还是带着 notion、飞书、石墨的惯性在用，会在意评论功能、多人编辑的实时协同、是否支持多维表格等等。

这就导致，基于本身缺少结构化信息的 md、csv、html 格式的本地文件，想做到 notion 文档、飞书表格的功能体验，是非常矛盾且困难的。我很早就提到了一个点是，agent 修改纯文本的 md 内容的手法，加上 git 的更新方式，和 yjs 这种为精细化实时协同而生的工具，是天然冲突的。但 moxt 就是，不用 yjs 但又希望有 yjs 的体验，不用 json 用 md、csv 但希望有评论、表格、链接等功能。

回到 moxt 产品上，现在为了让更多的用户上手，能有更好的体验，其实是在往一个旧时代的功能产品方向上打磨，但本身又是一个 agent first 的新东西，就有点拧巴。花了这么多时间精力，如果就是为了和 notion、飞书体验对齐，我对这件事不看好，能做的比人家这么多年积累出来的东西还好吗？即便能达到 80% 90%，又有什么意义呢？

既然是 agent first，为什么不能摒弃很多旧时代的 human 惯性，去完全做一个创新呢？我觉得创新的初期，很可能是不被大多数人看好和接受的，如果自己有足够的 vision 和 faith，坚持走一走试试，这才是大胆的创新。而且，如果能丢掉旧时代的产品包袱，新产品可能会轻量的多，迭代起来也会更快。

我认为这既是一个产品决策上的问题，也是一个技术团队的问题。我更新换在一个方向上做到专一且克制，做一个极致化的小而美的产品，而不是在初期就想着怎么做到各方面功能齐全、各种体验兼容的 all-in-one 的大而重的东西。这不是一个创业心态的团队做一个新产品的状态。
