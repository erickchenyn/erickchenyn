最开始我个人的 vibe coding 工作流中，是用 github issue 作为人与 agent、agent 之间异步沟通的上下文工具。在开发了 moxt 之后，因为 moxt 有持久化 context 和获取 repo 的能力，我尝试在 moxt workspace 中做类似的事，但我发现没有成功，原因可能有几点：

1. moxt workspace 的 context 与外界的工作流还是有些割裂，比如我的 coding 工作还是在本地 cc 中，cc 无法有效读写 moxt workspace context，即便有 moxt cli
2. moxt worksapce 中的 context 太多了，容易出现混乱，质量不高
3. moxt agent 比本地的 cc 还是要不稳定或慢一些，效率不高

综上，回到我个人的 vibe 工作流，除非完全不涉及 coding，我可以在 moxt workspace 中完成工作，但当涉及到 coding 时，我的工作流还是完全外置的，那么 context 就不能在 moxt 中了，github issue 还是那个最简单、干净的环境

我的思考是，context 的大而全不是最重要的，局部 context 的干净、隔离和 context 之间、context 与 agent 的沟通方式是最重要的，这使得每个工作流都可以灵活整合，而不是 all-in-one 在一个地方做完所有的事

即便像 claude 推出了 cowork、chrome 和 design 工具，它运行在某台机器上，我们还是有独立的手段来自由管理 context，context 不受 claude 产品的约束，还是开放的、可迁移的。这点就像我之前对于笔记 app 数据的思考一样，只有数据是开放的、可迁移的（本地优先，或至少可导出），数据格式是通用的（比如 markdown），数据本身才更有生命力更有价值
