# 使用 Claude Code：HTML 不可思议的有效性

> - 作者：Thariq（X @trq212）
> - 日期：2026-05-09
> - 原文链接：https://x.com/trq212/status/2052809885763747935

---

Markdown 已经成为 Agent 与我们沟通时使用的主流文件格式。它简单、可移植、具有一定的富文本能力，而且便于编辑。Claude 甚至已经出人意料地擅长在 Markdown 文件中使用 ASCII 来制作图表。

但随着 Agent 变得越来越强大，我感觉 Markdown 已经成为一种限制性的格式。我发现很难阅读超过一百行的 Markdown 文件。我想要更丰富的可视化、颜色和图表，并且希望能轻松分享它们。

我也越来越不亲自编辑这些文件了，而是将它们用作规格说明、参考文件、头脑风暴输出等。当我确实需要编辑时，通常是提示 Claude 来编辑它们，这消除了 Markdown 最大的优势之一。

我已经开始更倾向于使用 HTML 作为输出格式而非 Markdown，并且越来越多地看到 Claude Code 团队的其他人也在这样做，以下是原因。

（如果你想先看一些示例，可以在这里查看：https://thariqs.github.io/html-effectiveness，只是记得回来继续阅读为什么）

## 为什么选择 HTML？

### 信息密度

![信息密度](https://pbs.twimg.com/media/HHz_q48aAAAaCfW?format=jpg&name=large)

HTML 相比 Markdown 可以传达更丰富的信息。它当然可以做简单的文档结构如标题和格式化，但它还可以表示各种其他信息，例如：

- 使用表格展示表格数据
- 使用 CSS 展示设计数据
- 使用 SVG 展示插图
- 使用 script 标签展示代码片段
- 使用 HTML 元素配合 JavaScript + CSS 实现交互
- 使用 SVG 和 HTML 展示工作流
- 使用绝对定位和 Canvas 展示空间数据
- 使用 image 标签展示图片

我甚至可以说，几乎没有 Claude 能读取的信息集合是你不能用 HTML 相当高效地表示的。这使得它成为模型向你传达深度信息、供你审阅的一种高效方式。

我发现在无法这样做的情况下，模型可能会在 Markdown 中做一些更低效的事情，比如 ASCII 图表，或者我最喜欢的——用 Unicode 字符来估算颜色，就像这张来自 Claude Code 的截图一样。

![Claude Code 试图在 Markdown 中显示颜色](https://pbs.twimg.com/media/HH0CDc6a8AAy1bv?format=png&name=900x900)

### 视觉清晰度与易读性

![视觉清晰度](https://pbs.twimg.com/media/HH0AgqJbcAAaEcZ?format=jpg&name=large)

随着 Claude 能够完成更复杂的工作，它也在编写越来越大的规格说明和计划。实际上，我发现我往往不会真正阅读超过 100 行的 Markdown 文件，而且我肯定无法让组织中的其他人去读它。

但 HTML 文档更容易阅读，Claude 可以通过标签页、插图、链接等在视觉上组织结构使其便于导航。它甚至可以做成响应式的，让你根据不同的设备形态以不同方式阅读。

### 易于分享

Markdown 文件很难分享，因为大多数浏览器不能很好地原生渲染它们。你通常不得不将它们作为附件添加到邮件或消息中。

使用 HTML，只要你上传文件（例如到 S3），就可以轻松分享链接。你的同事可以在任何地方打开它并方便地引用。

别人真正阅读你的规格说明、报告或 PR 写作的可能性在 HTML 格式下要高得多。

### 双向交互

![双向交互](https://pbs.twimg.com/media/HH0Ao0tbYAAOF9e?format=jpg&name=large)

HTML 可以让你与文档交互，例如你可能想让它添加滑块或旋钮来调整设计，或者让你调整算法中的不同选项来看看会发生什么。你还可以让它允许你将这些更改复制为提示词，粘贴回 Claude Code。阅读我关于 playground 的帖子以查看这种双向交互的更多示例：https://x.com/trq212/status/2017024445244924382

数据摄取

为什么使用 Claude Code 制作 HTML 文件，而不是 ClaudeAI 或 Claude Design？最大的原因之一是 Claude Code 能摄取的所有上下文。例如，在写这篇文章时，我让 Claude Code 读取我的代码文件夹，找到我生成的所有 HTML 文件，对它们进行分组和分类，然后制作一个包含代表每种类型的所有图表的 HTML 文件。你在本文中看到的图表就是这样产生的。

除了文件系统，Claude Code 还可以通过你的 MCP（如 Slack、Linear 等）、你的网络浏览器（通过 Chrome 中的 Claude）、你的 git 历史等找到额外的上下文。

### 它是有趣的

用 Claude 制作 HTML 文档就是更有趣，让我感觉更投入和参与创作，仅此一点就足够了。

### 如何开始

我有点担心人们会读了这篇文章然后把它变成一个 /html 技能或什么的。虽然这可能有一些价值，但我想强调你不需要做太多就能让 Claude 做到这一点。你只需要让它"制作一个 HTML 文件"或"制作一个 HTML 产物"。

关键是知道你想让产物做什么以及你可能如何使用它。你可能随着时间推移会制作一个技能，但现在我建议从零开始提示，以掌握在不同情况下如何使用它。

## 使用场景

为了更具体，我为不同的使用场景制作了许多不同的 HTML 文件。你可以在这里查看所有示例：https://thariqs.github.io/html-effectiveness/，以下是概述。

### 规格说明、规划与探索

HTML 是 Claude 深入问题的丰富画布。当我开始处理一个问题时，我不会做一个简单的 Markdown 计划，而是期望制作一系列 HTML 文件。例如，我可能会先让 Claude Code 进行头脑风暴，创建一些不同选项的探索。然后我会让它深入展开其中一个，也许制作原型或代码片段。最后，当我感觉满意时，我会让它写一个实施计划。当我对计划满意时，我会创建一个新会话并传入所有这些文件让它去实现。

验证时，我也会让验证 Agent 读入这些文件，这样它就会对需要什么有更广泛的上下文。

![规格说明规划](https://pbs.twimg.com/media/HH0BFWLbMAEk_7T?format=jpg&name=large)

示例提示词：

- 我不确定入门引导屏幕该往什么方向走。生成 6 种明显不同的方案——变化布局、语调和密度——并在一个 HTML 文件中以网格形式排列，这样我可以并排比较。标注每个方案所做的权衡。
- 在一个 HTML 文件中创建一个详尽的实施计划，确保制作一些原型，展示数据流并添加我可能想审阅的重要代码片段。使其易于阅读和消化。

使用场景：

- 探索代码中的其他实现方式
- 探索多种视觉设计

### 代码审查与理解

代码在 Markdown 文件中很难阅读。但使用 HTML 我们可以渲染差异、注释、流程图、模块等。用它来理解 Agent 编写的代码，进行代码审查，或向审查你代码的人解释 PR。我发现这通常比默认的 GitHub diff 视图效果更好，我现在每个 PR 都附带一个 HTML 代码解释器。

![代码审查](https://pbs.twimg.com/media/HH0BRSQbMAAuuof?format=png&name=large)

示例提示词：

帮我审查这个 PR，创建一个描述它的 HTML 产物。我不太熟悉流式/背压逻辑所以重点关注那部分。渲染实际的 diff 并带有行内边注，按严重程度对发现进行颜色编码，以及其他可能需要的东西来很好地传达概念。

使用场景：

- 创建 PR
- 审查 PR
- 理解代码中的主题

### 设计与原型

Claude Design 基于 HTML，因为 HTML 在设计方面表达力极强，即使你最终的呈现面不是 HTML。Claude 可以用 HTML 勾勒设计，然后用你选择的语言编写，无论是 React、Swift 等。

你还可以原型化交互，如动画、操作等。考虑让 Claude 制作滑块、旋钮等来精确调整你想要的效果。

![设计原型](https://pbs.twimg.com/media/HH0BXqjboAAHGsw?format=jpg&name=large)

示例提示词：

我想原型化一个新的结账按钮，点击时播放一个动画然后快速变成紫色。创建一个 HTML 文件，包含几个滑块和选项让我尝试这个动画的不同选项，给我一个复制按钮来复制效果好的参数。

用途：

- 创建设计系统产物
- 调整组件
- 可视化组件库
- 原型化愉悦的动画

### 报告、研究与学习

Claude Code 非常擅长跨多个数据源综合信息并将其转换为可读的报告。你可以提示 Claude 搜索你的 Slack、代码库、git 历史、互联网等，并用它为你自己、领导层、团队等生成极具可读性的报告。

你可以将其组装成一个长 HTML 文档、一个交互式解释器，甚至是幻灯片/演示文稿。让 Claude 使用 SVG 制作图表来帮助可视化。例如，在我关于提示缓存的帖子中，我让 Claude 在阅读 git 历史后为我准备了一份关于我们所有提示缓存更改的深度 HTML 研究文件。

![报告研究](https://pbs.twimg.com/media/HH0Bp86bUAAJDyZ?format=jpg&name=large)

示例提示词：

我不理解我们的速率限制器实际上是如何工作的。阅读相关代码并生成一个 HTML 解释器页面：一个令牌桶流程图、3-4 个带注释的关键代码片段，以及底部的"注意事项"部分。为只读一次的人进行优化。

用途：

- 总结一个功能如何工作
- 向我解释一个概念
- 给老板的每周状态报告
- 给领导层的事故报告
- SVG 插图、流程图、技术图表等

## 自定义编辑界面

有时候纯粹在文本框中描述你想要什么很困难。在这种情况下，我会让 Claude 为我正在处理的确切事物构建一个一次性编辑器。不是产品，也不是可复用的工具，而是一个为这一份数据专门构建的单个 HTML 文件。

关键始终是以导出结束：一个"复制为 JSON"或"复制为提示词"按钮，将你在 UI 中做的任何事情转回可以粘贴到 Claude Code 中的内容。

![自定义编辑界面](https://pbs.twimg.com/media/HH0FbKebUAAsRPr?format=jpg&name=large)

示例提示词：

- 我需要重新排列这 30 个 Linear 工单的优先级。给我制作一个 HTML 文件，每个工单作为可拖拽的卡片在"现在 / 下一个 / 之后 / 砍掉"列之间移动。按你的最佳猜测预排序。添加一个"复制为 Markdown"按钮，导出最终排序及每个分类的一行理由。
- 这是我们的功能开关配置。为它构建一个基于表单的编辑器，按区域分组开关，显示它们之间的依赖关系，如果我启用了一个前置条件关闭的开关就警告我。添加一个"复制 diff"按钮，只给我更改的键。
- 我在调整这个系统提示词。制作一个并排编辑器：左边是可编辑的提示词，变量槽位高亮显示，右边是三个样本输入，实时重新渲染填充后的模板。添加字符/token 计数器和一个复制按钮。

用途：

- 重新排序、分类或分组任何事物（工单、测试用例、反馈）
- 编辑结构化配置（功能开关、环境变量、带约束的 JSON/YAML）
- 调整提示词、模板或文案并实时预览
- 策划数据集、批准/拒绝行、标记示例、导出选择
- 注释文档、转录或 diff 并导出注释
- 选择难以用文字表达的值：颜色、缓动曲线、裁剪区域、cron 表达式、正则表达式

### 常见问题

我一直在告诉很多人我如何转向了 HTML，也看到了一些重复出现的问题。

**它的 token 效率不是更低吗？**
虽然 Markdown 通常使用更少的 token，但我发现 HTML 增加的表达力以及我更高的阅读可能性意味着我获得了整体更好的输出。有了 Opus 4.7 的 100 万上下文窗口，增加的 token 使用在上下文窗口中并不真正明显。

**你现在什么时候还用 Markdown？**
老实说我几乎所有事情都已经停止使用 Markdown 了，但我可能是 HTML 极端主义者那一边的。

**我怎么查看 HTML 文件？**
我通常在本地用浏览器打开它（你可以让 Claude 打开它），或者如果你想要一个可分享的链接就上传到 S3。

**生成 HTML 不是比 Markdown 更慢吗？**
确实更慢！HTML 可能需要 Markdown 的 2-4 倍时间，但我发现结果是值得的。

**版本控制怎么办？**
这老实说是 HTML 最大的缺点之一，HTML 的 diff 相比 Markdown 噪音大且难以审查。

**如何让 Claude 匹配我的品味 / 不做得难看？**
前端设计插件帮助 Claude 制作好的 HTML 文件。但要匹配你自己公司的风格，你可以通过将 Claude 指向你的代码库来创建一个设计系统 HTML 文件。然后你可以使用该设计系统文件作为其他 HTML 文件的参考。

### 保持在循环中

以上所有都是想说，我认为我使用 HTML 的真正原因是我感觉与 Claude 的协作中更加"在循环中"。我曾经开始担心，因为我不再深入阅读计划，我将不得不简单地让 Claude 自己做决定。

但我很高兴地说，使用 HTML 后我比以前任何时候都更"在循环中"。希望你也是如此。
