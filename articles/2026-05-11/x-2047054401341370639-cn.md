# 多 Agent 系统：真正有效的实践

> - 作者：Walden（X @walden_yan）
> - 日期：2026-04-23
> - 原文链接：https://x.com/walden_yan/status/2047054401341370639

---

10 个月前，我写了[《不要构建多 Agent 系统》](https://cognition.ai/blog/dont-build-multi-agents)，论点是大多数人不应该尝试构建多 Agent 系统 [1]。并行 Agent 会对风格、边界情况和代码模式做出隐式选择。当时，这些决策经常互相冲突，导致产品脆弱。从那以后发生了很多变化。

在 Cognition，我们已经开始部署在实践中真正有效的多 Agent 系统。我们原来的观察对于并行写入型群集今天仍然成立：那个领域中大多数炫酷的想法仍然没有看到有意义的采用。但我们发现了一类更窄的确实有效的模式：多个 Agent 为一个任务贡献智力，而写入操作保持单线程的设置。在这篇文章中，我将总结我们在构建它们过程中学到的东西。

## 上下文工程回顾

在上一篇文章中，我们鼓励读者将 Agent 构建从"提示工程"重新定义为"上下文工程"。提示工程鼓励花哨的技巧，如"你是一位高级软件工程师"或"多想一会儿"。上下文工程更持久，专注于在假设模型会随时间变得更强的前提下，给模型提供正确的上下文。由于多种原因，上下文工程在多 Agent 设置中可能变得非常具有挑战性。过去，我们推荐了以下原则：

- **在 Agent 之间尽可能多地共享上下文。** 确保它们看到相同的信息来源，保持在同一页面（待办事项列表、计划文件），并共享关于它们要完成的整体任务的相同先验假设。如果需要，帮助它们进行通信。
- **行动携带隐式决策。** 当一个 Agent 做出某些更改或编辑时，它可能会做出隐式选择（风格、代码模式、某些边界情况应该如何处理），这些可能与其他并行 Agent 的隐式选择冲突。因此，在多个 Agent 采取写入操作的多 Agent 世界中，决策可能变得相当碎片化。

尽管过去几个月发生了很多变化，但对深思熟虑的上下文工程的需求并没有改变。作为原则 2 的推论，世界上大多数多 Agent 设置仅限于"只读"子 Agent，如网络搜索子 Agent 和代码搜索子 Agent。例如，Devin 可以调用 [Deepwiki](http://deepwiki.com/) 子 Agent 来获取代码库上下文。但这些类型的子 Agent 大多类似于工具调用，而非真正的多 Agent 协作。我们想探索当 Agent 以更交互的方式协作时可以解锁什么能力。

## 过去 10 个月发生了什么变化

首先，模型已经变得更加自然地"Agent 化"。它们直觉地理解工具使用、自身的上下文限制，以及如何为协作者（无论是人类还是其他）提炼上下文。结果是，Agent 的使用量增长了……巨量。即使我们看 Devin 在最大企业客户群体中的使用情况——这个群体传统上对采用新技术持谨慎态度——我们也看到过去 6 个月的爆发式增长（约 8 倍）。

![Devin 使用量增长](https://pbs.twimg.com/media/HGdPmr9bQAEow7_?format=jpg&name=large)

这种使用量的爆发对多 Agent 产生了推力和拉力。

在推力方面，能力的提升让用户自然地尝试了更多的多 Agent 设置。当你使用这么多 Agent 时，你自然会开始被 Agent 周围的一切事物瓶颈住：管理、规划和审查。例如，一些人创建了脚本让 Devin 管理其他 Devin。许多人也倾向于让编码 Agent 和审查 Agent 来回迭代。

在拉力方面，Agent 使用的爆发导致了成本的爆发。随着新的 [Mythos 级别](https://www-cdn.anthropic.com/08ab9158070959f88f296514c21b7facce6f52bc.pdf)更大、更强的模型即将到来，如何以更低成本实现前沿能力的自然问题就出现了。多 Agent 系统可能是一个自然的答案。

还有一波引人注目的演示，将大量 Agent 投入大型工程项目。著名的例子包括[构建网络浏览器](https://cursor.com/blog/scaling-agents)（20 万行代码）、[构建 C 编译器](https://www.anthropic.com/engineering/building-c-compiler)（10 万行代码）和[优化 LLM 训练脚本](https://github.com/karpathy/autoresearch)（1 万次以上迭代）。这些令人兴奋，但它们都共享一个大多数真实软件没有的属性：简单、可验证的成功标准。真实软件需要一个能扩展人类品味和决策的系统，这就是我们着手探索多 Agent 系统的背景。

## 一些实践性的多 Agent 实验

### 1）愚蠢到不应该有效的代码审查循环

你可能会认为让一个模型审查自己的代码不会产生任何有用的发现。但即使对于 Devin 编写的 PR，Devin Review 平均每个 PR 发现 2 个 bug，其中大约 58% 是严重的（逻辑错误、遗漏的边界情况、安全漏洞）。系统通常会循环多个代码审查周期，每次都发现新的 bug（这并不总是好事，因为可能需要一段时间）。今天，我们让 Devin 和 Devin Review 原生地相互迭代，这样大多数 bug 在人类打开 PR 时就已经解决了。

**反直觉的部分。** 有趣的是，我们发现这种技术在编码和审查 Agent 事先不共享任何上下文时效果最好。为什么？

这有哲学和技术方面的混合理由。首先，我们必须记住，即使 Agent 框架完全相同，将同一个模型放在两个 Agent 中，并不会像你想象的一个人同时做两个任务那样使它们产生自我偏见/相关性。这些 Agent 本质上是基于上下文执行的系统。它们没有自我意识，任何可能存在的共享偏见最终来自于它们的训练过程，我们现在可以假设训练质量相当高。

审查 Agent 拥有完全干净的上下文也帮助它深入原始编码 Agent 可能没有触及的领域。一方面，这是因为它被迫在没有规格说明的情况下从实现反向推理，并且可以公开质疑原始 Agent 由于用户指令错误而可能忽略的事情（例如，用户告诉 Agent 实现一个不安全的模式）。但更重要的是，拥有干净的上下文使 Agent 更聪明，因为注意力的数学原理。[Context Rot](https://www.trychroma.com/research/context-rot) 是一个有充分文献记录的现象，是模型在越来越长的上下文长度下做出越来越不智能的决策的结果。模型通常有有限数量的注意力头，当它们需要在不断增长的指令、提示、代码等上下文中工作时，重要细节可能不会被完全纳入其决策。当编码 Agent 在一个任务上工作了数小时，阅读代码库、运行命令、思考不同方法、修复错误时，它很快就积累了大量上下文。专用的审查 Agent 可以跳过这些多余的上下文，只看 diff，并在从头阅读代码时重新发现它需要的任何上下文。上下文更短，智能提升自然导致对细微问题的检测增加。

![代码审查循环](https://pbs.twimg.com/media/HGdPkDSa4AAGNqp?format=png&name=900x900)

使这个系统真正良好运作的最后关键部分是编码 Agent 和审查 Agent 之间的通信桥梁。基本上，Devin 是否正确地使用其更广泛的用户指令、决策等上下文来过滤从 Devin Review 返回的 bug？这是防止循环、违反用户意愿、做超出范围的工作等的关键。我们发现，通过一些专门的提示，今天的模型可以在这里做出合理的判断，你最终会得到两个 Agent 和人类之间一些非常有趣的互动。

![Agent 交互](https://pbs.twimg.com/media/HGdPYUtbMAAx_8l?format=jpg&name=large)

> 要点总结：在使用生成器-验证器循环时，干净的上下文会带来显著的能力提升。但清晰的通信和与整体上下文的综合对于连贯的体验很重要。

### 2）大型昂贵模型回归——介绍"Smart Friend"

如果你看看过去几个月最流行的模型，你会看到从 Anthropic 的 Sonnet 级中型模型到 Anthropic 的 Opus 级大型模型的明显转变，为了追求性能。随着 Mythos 的到来，我们基本可以说"规模化回来了"。

这其中悄然的含义是，前沿智能很快将对大多数日常任务来说太昂贵（也许太慢）。同时，使用较小模型你面临一个困境：任务可能比最初预期的更困难。

如何两全其美？在 Windsurf 中，我们带着这个目标尝试了一个实验，当我们在 10 月[发布 SWE-1.5](https://cognition.ai/blog/swe-1-5) 时，一个 950 token/秒的次前沿模型。我们发现，当与 Sonnet 4.5 搭配进行"规划"时，我们能够弥补一小部分性能差距，同时保持低成本和快速速度。

![Smart Friend 架构](https://pbs.twimg.com/media/HGdP0YlbcAAQZdA?format=png&name=240x240)

我们实际使用的架构是将更智能/更大的模型作为"smart friend"工具提供，主/较小模型可以调用它。基本上，让主/较小模型决定什么情况足够棘手，值得咨询更智能/更昂贵的模型。但我们很快发现，工程化上下文传递和通信很棘手：

**1. 主模型需要知道如何与 smart friend 对话。**

这种设置的核心棘手之处在于"一个更笨的模型如何知道它达到了极限？"与更流行的反向设置（智能主模型将任务委派给较小的子 Agent）不同，做出何时委派决定的模型不是更智能的那个。这里有几个潜在解决方案。首先，你可能鼓励主 Agent 总是至少调用一次智能 Agent 来评估是否有一些被遗漏的棘手之处。你也可能对主模型进行提示调优或训练，使其在这个决策上更加校准。根据主模型的智能程度，某些类型的领域特定指导性规定可能是必要的，比如在合并冲突时总是调用 smart friend。

与这种通信方法相关的另一个棘手问题是：主模型应该与 smart friend 共享什么上下文？此外，主模型应该问 smart friend 什么？如果主模型只共享其总上下文的一个子集，那么智能模型可能无法做出完全知情的决策。我们发现，对于今天的模型，一个合理的 80/20 解决方案是简单地与智能模型共享主模型完整上下文的一个分支。同样，我们发现鼓励主模型提出广泛的问题（"我应该怎么做？"）并让智能模型决定什么值得讨论会更好。

**2. Smart friend 需要知道如何回话给主模型**

无论你多好地调优 (1)，你可能仍会发现由于上下文丢失导致的质量差距。调优另一个方向的通信可以弥补这些差距。例如，假设主模型从未查看过 important_file.py，并向智能模型询问了需要了解此文件内容的事情。在这种情况下，智能模型的正确回答不是编造一些理论（这通常是默认行为），而是具体指示主模型调查这个文件并稍后再问。同样，让 smart friend 超越主模型所问的问题，基于 Agent 轨迹建议任何重要指导，即使主模型没有问，也通常是有益的。我们发现这种"超范围" smart friend 通常会带来更有趣的互动。

![Smart Friend 通信](https://pbs.twimg.com/media/HGdP7yPbsAA53Oi?format=png&name=900x900)

**Smart Friend 的实际结果**

我们应该坦诚：SWE 1.5 作为这种设置的主模型不够好。它与 Sonnet 4.5 之间的差距恰好在对这种设置最重要的地方太大了：知道何时升级、知道问什么。成本和速度的优势是真实的，但质量上限由主模型设定，而主模型不够强。SWE 1.6（一个[最近的后续版本](https://cognition.ai/blog/swe-1-6)，在 SWE-bench 上达到 Opus-4.5 级别的性能）明显更好，缩小了足够的差距使得这个模式开始有回报，但仍然不是我们想要的水平。我们相当确信这是一个训练问题，未来的 SWE 模型将在训练中考虑这种来回互动 [2]。

这个模式确实有效的地方是跨前沿模型之间。我们已经在生产中将 Claude 和 GPT 放在这种设置中运行了相当长一段时间，在最棘手的场景中产生了真实的收益。有趣的发现是，提示调优问题与小模型到大模型的情况不同。跨前沿通信不太关于较弱的模型知道何时向更强的模型求助，更多是关于路由到在特定子任务上最擅长的模型。有些模型调试更好，有些处理视觉推理更好，有些写测试更好。委派逻辑变成了能力路由器而非难度升级器。

> 要点总结：smart-friend 在两个模型都很强时今天就有效。让它在非对称较弱主模型的情况下工作——这是通向最大解锁的版本——仍然是一个开放问题，我们认为这是一个训练问题。如果你想交流心得，请联系我们。

## 展望：更高层级的委派

上面两种模式共享一个结构：一个写入者，由其他贡献智力的 Agent 围绕增强。显而易见的下一个问题是，这是否可以扩展到 Agent 拥有更大范围的所有权，例如，跨越十个 PR 的产品功能、涉及十几个服务的迁移、一周而非一个下午的工作。

这在今天的 Devin 中已经上线。一个管理者 Devin 可以将更大的任务分解成片段，生成子 Devin 来处理它们，并通过内部 MCP 协调它们的进度。让它感觉连贯比我们预期的需要更多的上下文工程。在小范围委派上训练的管理者默认过于规范化，当管理者缺乏深层代码库上下文时这会适得其反。Agent 假设它们与子代共享状态但实际上并没有。跨 Agent 通信——子 Agent 写消息回给管理者以传递给 Agent 团队中的其他 Agent——默认不会发生，因为模型没有在需要这样做的环境中被训练过。每个问题都需要专门的工作来修复，我们仍在改进所有这些。

**非结构化群集呢？** 我们认为非结构化群集方法——任意 Agent 网络相互协商——基本上是一种干扰。实际的形态是 map-reduce-and-manage：管理者拆分工作，子代执行，管理者综合并汇报。让这种系统感觉像单个 Agent 处理单个任务一样连贯，是我们 2026 年即将开展的一些工作的核心。

## 我们今天知道的

所有这些实验都有一条共同的主线：多 Agent 系统在写入保持单线程、额外 Agent 贡献智力而非行动时效果最好。干净上下文的审查者能发现编码者看不到的 bug。前沿级别的 smart friend 能发现较弱主模型遗漏的微妙之处。管理者跨子 Agent 协调范围而不碎片化决策。

开放问题都是通信问题。较弱的模型如何学会何时升级？子 Agent 如何展现应该改变其兄弟 Agent 工作的发现？如何在 Agent 之间传递上下文而不淹没接收者？你可以通过提示走得相当远，但我们也期望下一代模型，包括我们自己训练的模型，开始缩小这些差距。

我们正在构建一个世界，在软件开发生命周期的每个阶段注入智能——规划、编码、审查、测试和监控——不是作为一群自主的 Agent，而是作为一个扩展人类品味的协调系统。

欢迎在 [devin.ai](http://app.devin.ai/) 或 [windsurf.com](http://windsurf.com/) 试用我们的工作。如果你愿意与我们一起发现这些 Agent 构建原则，请联系 [walden@cognition.ai](mailto:walden@cognition.ai)。
