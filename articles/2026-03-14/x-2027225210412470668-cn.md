从 AWS Lambda 到 Unikraft 微虚拟机，搭配控制平面架构。

## 我们是怎么走到这一步的

我们在 Browser Use 上运行着数百万个 Web Agent。最初我们使用 AWS Lambda 运行仅浏览器的 Agent，每次调用都是隔离的，扩展是即时的，也不需要担心密钥泄露。

后来我们加入了代码执行功能。Agent 可以编写和运行 Python、执行 shell 命令、创建文件。我们将其构建为一个隔离的沙箱，Agent 作为工具来调用它。安全性没有问题：代码在沙箱中运行，而不是在后端。

但 Agent 循环仍然和 REST API 运行在同一个后端上。重新部署？所有运行中的 Agent 全部死亡。某个 Agent 占用大量内存？API 就会变慢。两个根本不同的工作负载共享同一个进程。

## 两种模式

当 Agent 可以运行任意代码时，它可以访问机器上的任何东西：环境变量、API 密钥、数据库凭证、内部服务。它需要与你的基础设施和密钥隔离。有两种方式可以做到这一点。

**模式 1：隔离工具。** Agent 运行在你的基础设施上。危险操作（代码执行、终端访问）在单独的沙箱中运行。Agent 通过 HTTP 调用沙箱。代码运行在一个没有任何可泄露内容的地方。

![模式 1 - 隔离工具。](https://pbs.twimg.com/media/HCIiADdbUAA0B0k?format=jpg&name=large)

**模式 2：隔离 Agent。** 整个 Agent 运行在一个零密钥的沙箱中。它通过持有所有凭证的控制平面与外界通信。

Agent 变成了一次性的。没有可窃取的密钥，没有需要保存的状态，你可以随时终止它、重启它、独立扩展它。控制平面掌握着真相。

![模式 2 - 隔离 Agent。](https://pbs.twimg.com/media/HCIiOKub0AEMvh2?format=jpg&name=large)

我们从模式 1 起步，后来转向了模式 2。

## 沙箱

同一个容器镜像在所有环境中运行。在生产环境中，它作为 [Unikraft](https://unikraft.cloud/) 微虚拟机运行。在本地开发和评估中，它作为 [Docker](https://www.docker.com) 容器运行。一个配置开关（`sandbox_mode: 'docker' | 'ukc'`）控制配置代码走哪条路径。

### 生产环境中的 Unikraft

每个 Agent 获得自己的 Unikraft 微虚拟机，启动时间不到一秒。我们通过 Unikraft Cloud 的 REST API 在 AWS 上的专用裸金属机器上进行配置。

沙箱从外部世界只接收三个环境变量：`SESSION_TOKEN`、`CONTROL_PLANE_URL` 和 `SESSION_ID`。没有 AWS 密钥，没有数据库凭证，没有 API 令牌。

Unikraft 开箱即用地提供了缩容到零的能力。当沙箱空闲时，虚拟机会挂起。当下一个请求到来时，它会恢复。在查询之间闲置的沙箱几乎不消耗资源，但对于后续任务会立即唤醒。

我们将沙箱分布在多个 Unikraft 区域，以防止任何单个区域成为瓶颈。

### 开发和评估中的 Docker

在本地和评估流水线中，同一个镜像作为 Docker 容器运行。相同的镜像、相同的入口点、相同的控制平面协议。我们可以在开发笔记本上运行完全相同的 Agent，为评估并行启动数百个，然后部署到 Unikraft 用于生产。

### 加固

沙箱在运行任何 Agent 代码之前会执行以下操作：

1. **仅字节码执行。** 在 Docker 构建期间，我们将所有 Python 源代码编译为 `.pyc` 字节码，然后删除所有 `.py` 文件。Agent 框架代码以 root 身份加载到内存中。加载后，源代码就消失了。
2. **权限降级。** 入口点以 root 身份启动（需要读取 root 拥有的字节码），然后立即通过 `setuid`/`setgid` 降级到 `sandbox` 用户。从那时起，所有操作都以非特权方式运行。
3. **环境变量清除。** 在将 `SESSION_TOKEN`、`CONTROL_PLANE_URL` 和 `SESSION_ID` 读入 Python 变量后，我们从 `os.environ` 中删除它们。如果 Agent 检查环境变量，这些变量已经不存在了。令牌在沙箱网络之外也没有用处。虚拟机位于私有 VPC 中，除了与控制平面通信外没有任何权限。

## 控制平面如何工作

可以将控制平面视为一个代理服务。沙箱无法直接访问外部世界。每个请求都必须经过控制平面。需要调用 LLM？经过控制平面。需要上传文件到 S3？经过控制平面。这是 Agent 与虚拟机外部任何东西通信的唯一方式。

它是一个无状态的 FastAPI 服务。来自沙箱的每个请求都携带 `Bearer: {session_token}` 头。控制平面通过令牌查找会话，验证其是否仍然活跃，然后使用真实凭证执行操作。

### LLM 代理

对于每次 LLM 调用，沙箱只发送新消息。控制平面在数据库中拥有完整的对话历史，在每次调用时重建它，并将完整上下文转发给提供商。这使沙箱保持无状态。你可以终止它并启动一个新的，对话会从中断的地方继续。

控制平面还负责执行成本上限和处理计费。沙箱只专注于任务本身。

### 通过预签名 URL 进行文件同步

沙箱有一个 `/workspace` 目录，Agent 在其中读写文件。文件同步服务监控变更并定期将它们同步到 S3，但沙箱永远看不到 AWS 凭证。相反，它向控制平面请求预签名 URL：

1. 沙箱检测到 `/workspace` 中的文件变更
2. 沙箱调用 `POST /presigned-urls`，传入文件路径
3. 控制平面生成预签名的 S3 上传 URL（限定在当前会话范围内）
4. 沙箱使用这些 URL 直接上传文件到 S3

下载以相反的方式工作。沙箱获得直接的、限定范围的 S3 访问权限，而无需持有任何 AWS 凭证。

### 网关协议

在沙箱内部，Agent 通过"网关"协议与控制平面通信：

```python
class AgentGateway(Protocol):
    async def invoke_llm(self, new_messages, tools, tool_choice) -> LLMResponse: ...
    async def persist_messages(self, messages) -> None: ...
```

在生产环境中，`ControlPlaneGateway` 向控制平面发送 HTTP 请求。对于本地开发和评估，`DirectGateway` 直接调用 LLM 并在内存中保持历史记录。Agent 代码不知道自己使用的是哪一个。相同的接口、相同的行为、不同的后端。

## 扩展

控制平面是无状态的：验证令牌、执行工作、返回结果。需要更多 Agent？启动更多沙箱。需要更多吞吐量？在负载均衡器后面添加控制平面实例。每一层都根据自己的瓶颈进行扩展。

我们的后端运行在 ALB 后面的私有子网中的 ECS Fargate 上。控制平面根据 CPU 利用率自动扩展。沙箱通过 Unikraft 独立扩展。每个会话都有自己的虚拟机，Unikraft 负责跨区域的调度。

## 总结

对于可以执行代码的 Agent，有两种沙箱化方式。你可以隔离工具（在沙箱中运行代码执行，将 Agent 保留在后端），或者隔离 Agent（将整个 Agent 放入沙箱，通过控制平面与外界通信）。

我们选择了模式 2。控制平面持有所有凭证，充当一切的代理：LLM 调用、文件存储、计费。沙箱接收三个环境变量，无法访问其他任何东西。它在生产环境中作为 Unikraft 微虚拟机运行，在开发和评估中作为 Docker 容器运行。所有环境使用相同的镜像。

代价是每个操作多一次网络跳转，以及需要部署三个服务而不是一个。实际上，与 LLM 响应时间相比，延迟可以忽略不计，而运维复杂度也是运维团队已经知道如何处理的那种。

**核心要点：你的 Agent 应该没有任何值得窃取的东西，也没有任何值得保留的东西。**
