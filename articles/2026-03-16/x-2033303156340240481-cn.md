# 规格即代码：Spec Driven Development 实践指南

> - 原文链接：https://x.com/juliandeangeIis/status/2033303156340240481

---

AI 编程 Agent 失败，不是因为模型太弱，而是因为指令太模糊、Agent 框架太脆弱。这就是为什么所有人都在构建规格（spec）和 Agent 框架。

在我之前那篇关于 [Coding Agent Harness](https://x.com/juliandeangeIis/status/2027888587975569534) 的文章中，我将 Spec Driven Development 介绍为"终极框架"，并承诺会深入展开。这篇就是那篇。

## 真正的瓶颈

关于 AI 编程 Agent，有一个令人不舒服的真相：最大的瓶颈不是模型、不是上下文窗口、也不是工具链。而是给出指令的那个人。

当人们刚开始使用编程 Agent 时，交互通常是这样的：

> "给后台添加一个管理条目的功能。"

Agent 读取代码库，选择一个模式，写出功能。乍一看还不错。然后你再次点击"添加条目"，它把同一条目插入了两次。

所有你以为显而易见的假设突然变得可见：操作应该是幂等的，只有管理员才能执行，"后台"应该是内部后台，而不是面向卖家的那个。

问题在于，这些都不在提示词里。Agent 不得不猜：哪个后台、哪个 API 契约、哪个存储层、哪个授权模型、哪种错误处理策略。

每一个都是一个无声的决策。有些猜对了，有些猜错了。随着复杂度的增加，你的意图和 Agent 实际构建的东西之间的差距也在扩大。

这就是模糊性问题。不是 Agent 的错，而是沟通问题。

## 为什么整个生态在收敛

整个生态系统正在独立地走向同一个解决方案。来看看正在发生的事情：

- **GitHub 的 Spec Kit（77k stars）**：构建了 spec-plan-task-implement 循环，与 Agent 无关，适用于 Claude Code、Cursor 等各种工具。
- **OpenAI 的 Symphony**：监控你的 Issue 追踪器，为每个 Issue 启动自主 Agent，并要求以 SPEC.md 作为契约。
- **Ralph Loop**：将 PRD 放入无限 Agent 循环中，进度通过文件和 Git 持久化，而不是上下文窗口。

就连编程 Agent 本身也在朝这个方向原生演进。

Claude Code 和 Cursor 中的 Plan 模式，本质上就是一个轻量级的 spec-and-plan 步骤。任务分解现在已经内置于大多数 Agent 循环中。

所有这些工具，尽管架构和理念各异，却共享同一个核心思想：**在写代码之前先定义你想要什么，然后让 Agent 从结构化规格中实现。**

这就是 Spec Driven Development 的核心思想。

## 什么是 Spec Driven Development？

Spec Driven Development（SDD）是一种方法论，而不是一个工具。它是一个简单的四步流程：

1. **Specify（规格）**：描述你想构建什么。
2. **Plan（计划）**：从技术层面规划如何构建。
3. **Break it down（拆解）**：分解成小的、有序的任务。
4. **Implement（实现）**：逐一交给 Agent 执行。

每一步都在减少模糊性。当 Agent 开始写代码时，它已经拥有了所需的一切：功能是什么、如何集成、边界情况是什么、测试应该验证什么、遵循什么架构。编程 Agent 不需要猜测。

当你把一个写好的 spec 和 plan 交给 Agent，你是在一次性工程化它的整个上下文窗口：架构决策、分步指导和验收标准，全部集中在一套 artifact 中。

最好的部分：你也应该用 Agent 来写 spec。

修改一个 spec 比从头写要容易得多。你还可以让 Agent 从同一个 spec 中找出模糊之处。

## Spec-Driven Development 的层级

并非所有的 SDD 实现都是平等的。在 [这篇文章](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html) 中，Birgitta Böckeler 将 spec 成熟度分为三个级别：

- **Spec-First（规格优先）**：在编码之前写 spec，但交付后丢弃。大多数人从这里开始，它已经消除了该周期的模糊性问题。
- **Spec-Anchored（规格锚定）**：spec 与代码一起存储在仓库中，并随之演进。Spec 成为团队的活文档。
- **Spec-as-Source（规格即源码）**：spec 是主要 artifact，你编辑 spec，代码被重新生成以匹配。这是前沿，我们还没有完全到达那里，但这是发展轨迹所指向的方向。

Spec Driven Development 不是非此即彼。它是一种渐进式方法论。从第一级开始，获取价值，然后决定是否需要深入。

现在让我们分解 SDD 流程的每个步骤。

## Spec：描述什么，而非如何

Spec 是功能层。它描述功能做什么，而不是如何实现。它有意地与技术无关。

一个好的 spec 以非技术语言定义功能的目的、用例、需求、边界情况和成功标准。

关键洞察：**将功能与技术分离，可以减少 LLM 的不确定性。**

当你把"用户可以用 Google 和 GitHub 认证"和"使用 NextAuth.js 的 JWT 策略，将 session 存储在 Redis 中"混在一起时，你是在强迫 Agent 同时处理两种不同的关切。

功能需求可能是清晰的，但技术决策会创造出分支路径，复合了模糊性。

通过保持 spec 的纯粹**功能性**，你给了 Agent 一个清晰的目标，而不用过早的实现决策来污染它。你还从一开始就定义了边界情况的预期行为，在任何人写一行代码之前。

验收标准使用 Given/When/Then 格式来使验证无歧义：

> Given 一个新用户，When 他们点击"使用 Google 登录"并授权应用，Then 他们被重定向到带有有效 session 的 dashboard。

> Given 一个已有 Google 账户的用户，When 他们尝试用同一邮箱的 GitHub 账户登录，Then 账户被关联，他们访问同一个 profile。

这些不只是文档，它们成为**测试计划**。Agent 可以根据它们**验证**自己的实现。

## Plan：开发者发挥专业能力的地方

Plan 是**技术层**。它是 Agent 的实现指南：如何实现 spec 描述的一切。

这是**开发者的专业知识最重要**的地方。不是在写代码上，而是在做架构决策上：

- **架构和技术决策**："使用带有 App Router 的 NextJS。遵循 @auth-rules 中现有的认证模式。"
- **数据模型和契约**："使用现有的 User 实体，添加 provider 字段。"
- **测试策略**："单元测试必须覆盖 90% 的认证流程。"
- **性能约束**："登录 endpoint 的响应时间在 100ms 以内。"

Plan 是你引用自定义规则、指向代码库中现有模式、指定 Agent 应使用哪些 MCP、以及设置技术约束的地方。

它将抽象的 spec 转化为一个**具体的**、有边界的**实现指南**。

有一点让这个方法特别有效：plan 可以利用自动研究和你当前的 Agent 框架。

Spec Kit 的 `/plan` 命令等工具会在生成技术计划之前读取你的实际代码库。它分析你当前的结构，识别模式和约定，并提出与现有结构一致的架构。

## Tasks：分而治之

Plan 被分解为小的、有序的任务。每一个都应该可以在单个 Agent session 中完成，并产生一个有测试的可验证变更。

关键：每个任务必须是**自包含且无歧义的**。

Agent 不应该需要做假设或搜索缺失的上下文。如果一个任务需要 Agent 猜测，它就还没有准备好。

这也是开发者审查方案的地方。你的工作是**检查**任务列表是否**过度工程化**了解决方案。

真的需要七个任务吗，还是三个就够了？Agent 是否在创建不必要的抽象？任务分解是代码开始编写之前的最后一个检查点，所以值得花一分钟确保计划是务实的。

一旦任务确定，它们就解锁了两个强大的能力：并行性和 Agent 无关性。

独立的任务可以由多个 Agent 同时执行。

由于任务是自包含的，所有上下文都嵌入其中，你可以**随时切换 Agent**。

从 Claude Code 开始，用 Cursor 接手一个任务，用 Codex 完成另一个。在 Spec Kit 等良好框架的支撑下，spec 和 plan 随任务一起传递，而不是随 Agent。

## 权衡

Spec Driven Development 不是免费的。

spec-plan-task 循环消耗（大量）token。一个完整的 SDD session 可能比直接提示 Agent 多消耗 2-3 倍或更多的 token。这是权衡：你预先花更多，换来**显著更好的结果**。

在当前状态下，对所有事情都用它也不合适，因为你还必须手动指定每个步骤。

小变更、快速 bug 修复、配置更新——这些不需要完整的 spec。对于这些，使用 Plan 模式或直接提示即可。

SDD 在功能足够复杂、模糊性会导致 Agent 跑偏时大放异彩：多文件变更、涉及多个领域的功能、遗留代码库，任何"显而易见"的业务逻辑实际上并不显而易见的情况。

还有一条**学习曲线**。

开发者需要从"描述我想要的代码"转变为"描述我需要的行为"。这是一种心态转变。

## 我们在 MercadoLibre 如何推广

在 MercadoLibre，我们正在向近 2 万名开发者推介 SDD。在这个规模下有两个主要挑战。

第一个是习惯改变。开发者习惯于直接跳入代码。要求他们先写 spec 感觉像是额外负担——直到他们看到结果。

我们很早就发现，教授 SDD 价值的最好方式不是解释，而是**实践**。这就是为什么我们全力投入动手工作坊，目前已有超过 5000 名开发者参加。

开发者在一个 session 中写 spec、生成 plan、拆解任务、用 Agent 实现——这种体验推动了采用。

第二个是方法论周边的上下文。SDD 本身还不够，Agent 还需要你的内部工具、SDK 和平台约定的上下文。这就是我上篇文章中的 Agent 框架：编码你的标准的自定义规则、捆绑领域知识的 skill，以及连接内部系统的 MCP。

方法论定义了要构建什么。框架给了 Agent 正确构建它的内部上下文。

## 从哪里开始

如果这篇文章只有一个收获：**试试 Plan 模式。**

这是 SDD 最容易上手的入口。你可以获得大部分好处（减少模糊性、技术概要、更清晰的意图），而不需要完整的仪式感。很好的起点。

当你准备好更进一步时，采用专用的 SDD 框架，如 Spec Kit。它为 spec、plan 和 task 提供了结构化的流程，随着实现规模的增长，比临时提示要好扩展得多。

开始之前你必须知道：SDD 的价值随着实践而复利增长。你写的第一个 spec 会感觉缓慢而不完美。

但随着时间的推移，你会培养出一种直觉：该规格什么、什么时候是在过度设计 spec、Agent 真正需要什么才能做对。这种技能在每次迭代中都在积累。
