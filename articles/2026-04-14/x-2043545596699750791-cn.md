# 为什么你的"AI 优先"战略可能是错的

> - 作者：Peter Pang（X @intuitiveml）
> - 日期：2026-04-13
> - 原文链接：https://x.com/intuitiveml/status/2043545596699750791

---

![封面](https://pbs.twimg.com/media/HFwEJl_bEAAPyc8?format=jpg&name=large)

我们 99% 的生产代码都是由 AI 编写的。上周二，我们上午 10 点上线了一个新功能，中午完成了 A/B 测试，下午 3 点因为数据表现不佳而下线了它。下午 5 点我们就上线了一个更好的版本。三个月前，这样一个周期需要六周时间。

我们并不是靠在 IDE 里加个 Copilot 就做到这一切的。我们拆掉了整个工程流程，围绕 AI 重新构建了它。我们改变了规划、构建、测试、部署和团队组织的方式。我们改变了公司里每个人的角色。

CREAO 是一个 Agent 平台。25 名员工，10 名工程师。我们从 2025 年 11 月开始构建 Agent，两个月前我从零开始重构了整个产品架构和工程工作流。

OpenAI 在 2026 年 2 月发表了一个概念，恰好描述了我们一直在做的事情。他们称之为 harness engineering（工具链工程）：工程团队的首要任务不再是写代码，而是让 Agent 能够完成有用的工作。当某件事失败时，解决方案永远不是"更努力地尝试"，而是：缺少什么能力，以及如何让它对 Agent 来说是可理解和可执行的？

我们自己得出了同样的结论。只是当时没有给它起名字。

## AI 优先不等于使用 AI

![AI 优先不等于使用 AI](https://pbs.twimg.com/media/HFwEVlnbkAAYtWM?format=jpg&name=large)

大多数公司只是把 AI 嫁接到现有流程上。工程师打开 Cursor，PM 用 ChatGPT 写需求文档，QA 尝试用 AI 生成测试。工作流保持不变。效率提升了 10% 到 20%。没有结构性改变。

那叫 AI 辅助。

> AI 优先意味着你要围绕"AI 是主要构建者"这一假设，重新设计你的流程、架构和组织。你不再问"AI 如何帮助我们的工程师？"，而是开始问"我们如何重构一切，让 AI 来构建，工程师提供方向和判断？"

差异是倍数级的。

我看到一些团队声称自己是 AI 优先的，但依然在跑同样的 Sprint 周期、同样的 Jira 看板、同样的每周站会、同样的 QA 签收流程。他们把 AI 加入了循环，但并没有重新设计这个循环。

一个常见的版本就是人们所说的 vibe coding（氛围编程）。打开 Cursor，不断提示直到代码能用，提交，重复。这能产出原型。但一个生产系统需要是稳定的、可靠的、安全的。你需要一个系统来保证这些属性——即使代码是 AI 写的。你构建系统，提示词是一次性的。

## 为什么我们必须改变

去年，我观察团队的工作方式，发现了三个会把我们拖垮的瓶颈。

**产品管理瓶颈**

我们的 PM 花数周时间进行调研、设计、编写功能规格。产品管理几十年来都是这样工作的。但 Agent 可以在两小时内实现一个功能。当构建时间从数月缩短到数小时，数周的规划周期就成了瓶颈。

花几个月时间思考一个东西，然后两小时就建好了——这不合理。

PM 需要进化为以产品思维为导向的架构师，以迭代的速度工作，否则就要退出构建周期。设计需要通过快速的"原型-上线-测试-迭代"循环来完成，而不是在委员会中审查需求文档。

**QA 瓶颈**

同样的问题。Agent 完成一个功能后，我们的 QA 团队花好几天测试边界情况。构建时间：两小时。测试时间：三天。

我们用 AI 构建的测试平台替代了手动 QA，用来测试 AI 编写的代码。验证必须和实现保持同样的速度。否则你只是在老瓶颈下游十英尺的地方又造了一个新瓶颈。

**人力瓶颈**

我们的竞争对手有 100 倍甚至更多的人在做类似的工作。我们只有 25 人。我们无法通过招聘来追平差距，只能通过重新设计来实现。

三个系统需要 AI 贯穿其中：产品设计、产品实现和产品测试。任何一个环节保持手动，就会制约整条流水线。

## 大胆的决定：统一架构

![统一架构](https://pbs.twimg.com/media/HFwEjnpbEAAXeFc?format=jpg&name=large)

我必须先修好代码库。

我们的旧架构分散在多个独立系统中。一个变更可能需要改动三四个代码仓库。从人类工程师的角度看，这可以管理。从 AI Agent 的角度看，这是不透明的。Agent 看不到全貌，无法推理跨服务的影响，也无法在本地运行集成测试。

我必须把所有代码统一到一个 monorepo 中。一个原因：让 AI 能看到一切。

这就是 harness engineering 原则的实践。你把系统中越多的部分拉入 Agent 可以检查、验证和修改的形式，你就能获得越多的杠杆。分散的代码库对 Agent 来说是不可见的。统一的代码库是可读的。

我花了一周设计新系统：规划阶段、实现阶段、测试阶段、集成测试阶段。然后又花了一周用 Agent 重新架构整个代码库。

CREAO 是一个 Agent 平台。我们用自己的 Agent 重建了运行 Agent 的平台。如果产品能构建自己，那它就是有效的。

## 技术栈

以下是我们的技术栈及其作用。

**基础设施：AWS**

我们运行在 AWS 上，使用自动扩缩容的容器服务和熔断回滚机制。如果部署后指标恶化，系统会自动回退。

CloudWatch 是中枢神经系统。所有服务的结构化日志，超过 25 个告警，自定义指标每天被自动化工作流查询。每一个基础设施组件都暴露结构化的、可查询的信号。如果 AI 读不了日志，它就无法诊断问题。

**CI/CD：GitHub Actions**

每个代码变更都要通过六阶段流水线：

验证 CI → 构建并部署 Dev → 测试 Dev → 部署 Prod → 测试 Prod → 发布

每个 Pull Request 的 CI 关卡强制执行类型检查、Lint、单元测试和集成测试、Docker 构建、Playwright 端到端测试以及环境一致性检查。没有阶段是可选的。没有手动覆盖。流水线是确定性的，因此 Agent 可以预测结果并推理故障。

**AI 代码审查：Claude**

每个 Pull Request 触发三个并行的 AI 审查流程，使用 Claude Opus 4.6：

- 第 1 轮：代码质量。逻辑错误、性能问题、可维护性。
- 第 2 轮：安全性。漏洞扫描、认证边界检查、注入风险。
- 第 3 轮：依赖扫描。供应链风险、版本冲突、许可证问题。

这些是审查关卡，不是建议。它们与人工审查并行运行，在大量 PR 中捕捉人类遗漏的问题。当你每天部署八次时，没有人类审查者能在每个 PR 上保持专注。

工程师还可以在任何 GitHub Issue 或 PR 中 @claude 来获取实现方案、调试会话或代码分析。Agent 能看到整个 monorepo。上下文在对话之间保持连贯。

**自愈反馈循环**

这是核心。

每天早上 UTC 时间 9:00，一个自动化健康检查工作流运行。Claude Sonnet 4.6 查询 CloudWatch，分析所有服务的错误模式，并生成一份执行级健康摘要，通过 Microsoft Teams 发送给团队。无需任何人主动请求。

一小时后，分诊引擎运行。它将 CloudWatch 和 Sentry 的生产错误聚类，按九个严重性维度为每个集群评分，并在 Linear 中自动生成调查工单。每个工单包含示例日志、受影响用户、受影响端点以及建议的调查路径。

系统会去重。如果一个已开启的 Issue 覆盖了相同的错误模式，它会更新该 Issue。如果一个已关闭的 Issue 再次出现，它会检测到回归并重新打开。

当工程师推送修复后，同一条流水线处理它。三轮 Claude 审查评估 PR。CI 验证。六阶段部署流水线在每个阶段进行测试，逐步推进到 dev 和 prod。部署后，分诊引擎重新检查 CloudWatch。如果原始错误已解决，Linear 工单自动关闭。

![自愈反馈循环](https://pbs.twimg.com/media/HFwUNbua4AA65-z?format=jpg&name=large)

每个工具处理一个阶段。没有工具试图做所有事情。每日循环创建了一个自愈循环，其中错误被检测、分诊、修复和验证，几乎不需要人工干预。

我对 Business Insider 的一位记者说过："AI 会创建 PR，人类只需要审查是否存在风险。"

**功能开关和配套工具栈**

Statsig 管理功能开关。每个功能都在开关后面上线。发布模式：先对团队启用，然后逐步按百分比发布，最后全量发布或关闭。关闭开关可以即时下线一个功能，无需部署。如果一个功能导致指标下降，我们在几小时内就会撤掉它。不好的功能上线当天就会被淘汰。A/B 测试通过同一系统运行。

Graphite 管理 PR 分支：合并队列将 PR rebase 到 main，重新运行 CI，只有绿色才合并。堆叠 PR 允许高吞吐量的增量审查。

Sentry 报告所有服务的结构化异常，通过分诊引擎与 CloudWatch 合并以获得跨工具的上下文。Linear 是面向人类的层：自动创建的工单带有严重性评分、示例日志和建议的调查方向。去重防止噪音。后续验证自动关闭已解决的问题。

## 一个功能从想法到生产的流程

![功能流水线](https://pbs.twimg.com/media/HFwUDbna8AAQ8_F?format=jpg&name=large)

**新功能路径**

1. 架构师将任务定义为一个结构化提示，包含代码库上下文、目标和约束。
2. Agent 分解任务、规划实现、编写代码并生成自己的测试。
3. PR 打开。三轮 Claude 审查评估它。人类审查者检查战略风险，而非逐行正确性。
4. CI 验证：类型检查、Lint、单元测试、集成测试、端到端测试。
5. Graphite 的合并队列 rebase，重新运行 CI，绿色则合并。
6. 六阶段部署流水线在每个阶段进行测试，逐步推进到 dev 和 prod。
7. 功能开关为团队打开。逐步按百分比发布。监控指标。
8. 如有恶化可使用关闭开关。严重问题触发熔断器自动回滚。

**Bug 修复路径**

1. CloudWatch 和 Sentry 检测到错误。
2. Claude 分诊引擎评估严重性，在 Linear 中创建带有完整调查上下文的 Issue。
3. 工程师进行调查。AI 已经完成了诊断。工程师验证并推送修复。
4. 同样的审查、CI、部署和监控流水线。
5. 分诊引擎重新验证。如已解决，工单自动关闭。

两条路径使用同一条流水线。一个系统，一个标准。

## 成果

![成果](https://pbs.twimg.com/media/HFwUohKbcAAi0cm?format=png&name=large)

在 14 天内，我们平均每天进行三到八次生产部署。在我们的旧模式下，整个两周时间甚至无法产出一次生产发布。

不好的功能上线当天就被撤掉。新功能在构思当天就上线。A/B 测试实时验证效果。

人们以为我们是在用质量换速度。用户参与度上升了。支付转化率上升了。我们产出了比以前更好的结果，因为反馈循环更紧密了。每天发布比每月发布能学到更多。

## 新的工程组织

将存在两种类型的工程师。

**架构师**

一到两个人。他们设计标准操作规程，教 AI 如何工作。他们构建测试基础设施、集成系统、分诊系统。他们决定架构和系统边界。他们定义什么是 Agent 眼中的"好"。

这个角色需要深度的批判性思维。你批评 AI，而不是跟随它。当 Agent 提出一个方案时，架构师找出漏洞。它遗漏了什么故障模式？它越过了什么安全边界？它在积累什么技术债务？

我有物理学博士学位。我的博士教会我的最有用的东西是如何质疑假设、压力测试论点以及寻找遗漏之处。批评 AI 的能力将比编写代码的能力更有价值。

这也是最难填补的角色。

**操作者**

其他所有人。工作很重要，但结构不同了。

AI 给人类分配任务。分诊系统发现 Bug、创建工单、呈现诊断结果并分配给合适的人。此人调查、验证并批准修复。AI 创建 PR。人类审查是否存在风险。

这些任务是 Bug 调查、UI 精调、CSS 改进、PR 审查、验证。它们需要技能和专注，但不需要旧模式所要求的架构性推理。

**谁适应得最快**

我注意到了一个出乎意料的模式。初级工程师比高级工程师适应得更快。

传统实践较少的初级工程师感到被赋能了。他们拥有了能放大其影响力的工具。他们没有十年的习惯需要去除。

传统实践很强的高级工程师适应起来最困难。他们两个月的工作量可以被 AI 在一小时内完成。在花了多年时间培养稀有技能之后，这很难接受。

我不是在做评判。我在描述我观察到的现象。在这次转型中，适应能力比积累的技能更重要。

## 人性的一面

**管理的崩塌**

两个月前，我 60% 的时间花在管理人上。对齐优先级、开会、给反馈、辅导工程师。

今天：不到 10%。

传统的 CTO 模式说要赋能团队做架构工作，培训他们，授权他们。但如果系统只需要一两个架构师，我需要先自己来。我从管理转向了构建。大多数日子我从早上 9 点写代码到凌晨 3 点。我设计系统的 SOP 和架构。我维护工具链。

压力更大了。但我享受构建，而不是对齐。

**更少的争论，更好的关系**

我和联合创始人及工程师的关系比以前更好了。

转型前，我和团队的大部分互动是对齐会议。讨论权衡、辩论优先级、在技术决策上产生分歧。这些对话在传统模式中是必要的，但也很消耗人。

现在我仍然和团队交谠。我们聊其他事情。非工作话题、闲聊、团建旅行。我们相处得更好了，因为我们不再为那些系统可以轻松完成的工作争论不休。

**不确定性是真实的**

我不会假装每个人都很开心。

当我不再每天和人们交谈时，一些团队成员感到不安。CTO 不再和我说话意味着什么？在这个新世界里我的价值是什么？这些都是合理的担忧。

有些人花在辩论 AI 能否做他们工作上的时间比实际做工作还多。过渡期会产生焦虑。我没有一个干净的答案。

但我有一个原则：我们不会因为工程师引入了生产 Bug 就解雇他。我们改进审查流程，加强测试，增加防护栏。对 AI 也一样。如果 AI 犯了错，我们构建更好的验证、更清晰的约束、更强的可观测性。

## 超越工程

我看到其他公司采用了 AI 优先的工程，却把其他一切都保持手动。

如果工程在几小时内发布功能，但营销需要一周才能宣传，那营销就是瓶颈。如果产品团队仍然运行月度规划周期，规划就是瓶颈。

在 CREAO，我们将 AI 原生运营推进到了每个职能：

- 产品发布说明：从变更日志和功能描述中 AI 生成。
- 功能介绍视频：AI 生成的动态图形。
- 社交媒体日常发帖：AI 编排并自动发布。
- 健康报告和分析摘要：从 CloudWatch 和生产数据库中 AI 生成。

工程、产品、营销和增长在一个 AI 原生工作流中运行。如果一个职能以 Agent 速度运行，而另一个以人类速度运行，人类速度的职能就会制约一切。

## 这意味着什么

**对工程师**

你的价值正在从代码产出转向决策质量。快速编写代码的能力每个月都在贬值。评估、批评和指导 AI 的能力则在升值。

产品感觉或品味很重要。你能在用户告诉你之前就看出一个生成的 UI 是错的吗？你能看出 Agent 遗漏的故障模式吗？

我告诉我们 19 岁的实习生：训练批判性思维。学会评估论点、发现漏洞、质疑假设。学会什么是好的设计。这些技能会复利增长。

**对 CTO 和创始人**

如果你的 PM 流程比构建时间还长，从那里开始。

在扩展 Agent 之前先构建测试工具链。没有快速验证的快速 AI 就是快速移动的技术债务。

从一个架构师开始。一个人构建系统并证明它有效。系统运行后再让其他人以操作者角色加入。

将 AI 原生推进到每个职能。

预期会有阻力。有些人会反对。

**对行业**

OpenAI、Anthropic 以及多个独立团队殊途同归，收敛到了相同的原则：结构化上下文、专业化 Agent、持久记忆和执行循环。Harness engineering 正在成为标准。

模型能力是驱动这一切的时钟。我将 CREAO 的整个转变归因于过去两个月。Opus 4.5 做不到 Opus 4.6 能做的事。下一代模型将进一步加速。

我相信一人公司会变得普遍。如果一个架构师加上 Agent 能完成 100 人的工作，许多公司将不需要第二个员工。

## 我们还处于早期

我交谈过的大多数创始人和工程师仍然按传统方式运作。有些人在考虑转型。真正做到的人很少。

一位记者朋友告诉我，她就这个话题采访了大约五个人。她说我们比任何人都走得更远："我不认为有人像你们这样彻底重建了整个工作流。"

任何团队都可以用这些工具做到这一点。我们的技术栈中没有任何专有的东西。

竞争优势在于围绕这些工具重新设计一切的决心，以及承受代价的意愿。代价是真实的：员工的不确定感、CTO 每天工作 18 小时、高级工程师质疑自己的价值、两周时间里旧系统已经消失而新系统尚未验证。

我们承受了这个代价。两个月后，数字说明了一切。

我们构建一个 Agent 平台。我们用 Agent 构建了它。
