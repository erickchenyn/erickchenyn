> - 原文链接：https://x.com/hwwaanng/status/2053830423331865077

![](https://pbs.twimg.com/media/HICdVzdagAAyEzU?format=jpg&name=large)

Claude Code 在 4 月份重写了它的桌面应用，发布之后舆论分成了两半。

![](https://pbs.twimg.com/media/HICruuUa4AAWqSG?format=jpg&name=large)

很多人最直接的反应是：怎么看起来更"廉价"了？图标更少，留白更多，颜色更冷，整体调子像一个还没做完视觉的 alpha 版工程工具。

> "看起来更简陋"并没有问题，Claude Code 的新 UI，确确实实，缩小了字体，减少了 icon 的使用，减少了页面拥有的颜色。并且 Chat / Cowork / Code 都一起改变了。

但为什么呢？

## 那些新增的部分

要回答这个问题，我们先看看 Anthropic 的产品团队，给 Claude Code 添加了什么新功能。

新的桌面 Claude Code 把整个界面重组成几样东西：

左边一条 sidebar 集中管理多个 session，可按 status、project、environment 过滤，PR 合并或关闭后 session 会自动归档（你可以在 Settings 里面设置）；

右边是一组可拖拽的 pane，terminal、preview、diff viewer、chat、文件编辑器随你拼成网格；

新增了一个叫 side chat 的旁路对话（/btw），用 ⌘+; 打开，它从主线程读取上下文，但不写回主线程；信息密度被切成了 verbose、normal、summary 三档，由你自己决定要看到 Claude 的多少推理过程；

底部有一个 usage 按钮，同时显示 context window 用量和 session 用量。

> [claude code desktop redesign](https://claude.com/blog/claude-code-desktop-redesign)

![](https://pbs.twimg.com/media/HICeIXpbYAAFHLF?format=jpg&name=large)

把这些细节连起来看，新 UI 的重心已经从聊天框挪到了**任务面板**。Anthropic 自己在公告里把用户的位置定义为 "orchestrator seat"——你成了那个同时管多个并行任务的调度者，产品上，Anthropic 似乎在强调用户应该是围绕"调度"在使用产品。

"调度"，添加了很多产品的复杂性。

![](https://pbs.twimg.com/media/HICmdQbbUAApLmb?format=jpg&name=large)

我相信，很多人并没有注意到，Anthropic 的 Code 已经有如此之多的复杂功能了。

事实上 Anthropic 把这些 UI 都做成了一个模样：更克制、更结构化、视觉更冷的样子。

为什么不能是一个花哨一点的调度界面？

## 为什么 AI 真的需要这种 UI

答案很简单，Anthropic 自己的模型，不太会写 UI —— 而这条事实正在反过来决定了产品该长什么样。

一篇被广泛流传博客《Every AI Coding Agent Is Blind to Your UI》直接点名 Claude Code、Cursor、Copilot、Windsurf——所有 file-first 的 AI coding agent 共享同一个 blind spot：它们能读源代码、追导入、跑 grep，但看不见运行时。具体来说，computed style、组件 ID、responsive 分支、stacking context、hover/focus/animation 这些东西**都不在源代码里**。它们只存在于浏览器渲染之后那一瞬间。

> https://frontman.sh/blog/introducing-frontman/

作者把这件事定义成 "architectural gap"，并下了一个相当强硬的判断：模型变强不能修这个问题。GPT-5 不会修，Claude 下一代也不会修，因为它们读的都是同一份不包含运行时信息的源代码。

所以，Anthropic 是在做的 "AI-friendly UI"：

- 视觉噪音越少，agent 越不容易在不该改的地方乱改；
- 功能分区越清楚（sidebar、pane、editor、preview），它越知道自己在哪个语境里工作；
- 状态越显式（context 用量、session 状态、diff），它越能照着事实做决定，少靠猜；
- 上下文越能被人为隔离（side chat 那种结构），主任务的推理链越不容易被临时问题污染。

结构上，每一个 panel 都像是独立的「小 App」：

> 正常的数据结构，分层的抽象，实现一个功能往往是一个功能需要涉及到不同数据层的不同文件。合理的架构能够把不同业务抽象整合起来，让项目质量和需求迭代都有一个好的结果。而 Vibe coding，需要让一个 feature 的和另一个 feature 的文件少交互，这样 AI 才能更好地工作。
>
> — Hwang (@hwwaanng), [May 26, 2025](https://x.com/hwwaanng/status/1926845700865692024)

读到这里很容易以为我在反对"漂亮 UI"。其实更像是漂亮 UI 在新前提下被重新校准了一遍：复杂浮层、阴影叠加、隐式动画状态对人类用户是细节，对 agent 是认知噪音。当你的产品里多了一个"看不见网页的同事"，UI 优化目标被重新加权了。

## "Seeing like an agent"

模型 "看不见"，所以 UI 必须为它减压。Anthropic 内部确实在用"从模型视角设计"这条原则做产品。

> 另一篇博客里。Anthropic 在 4 月 10 日——也就是 UI 重设计上线前的 4 天——发了一篇工程博客，标题就叫《Seeing like an agent: how we design tools in Claude Code》。它讲的是 Claude Code 团队在设计 tools 时的方法论：从模型自己的角度去想，它在哪些信息上有视野、在哪些上没有，哪些工具一开始有用、模型变强之后反而成了束缚。

文章最后用了一句很直接的话——"try to see like an agent"。

> https://claude.com/blog/seeing-like-an-agent

这条方法论原本只用来解释 tool 设计：要不要做 AskUserQuestion、Task tool、Grep。我承认这是推断，产品的设计哲学很少明文写在公告里，它通常藏在两件事之间的相似性里。约束在外，意图在内，刚好对得上。See like an agent。

## 启发

减少页面不必要的样式，明确足够精炼的规则，限定 AI 的自主发挥，就可以让你和 Agent 有更好的协同，让团队的迭代更高效。

不过，也许软件行业也会因此更加无趣。

## 审美还重要吗

一年前，人们还在 diss 蓝紫色渐变是 AI Slop，今天人们说类似 Anthropic 米色是新的 AI Slop。

> this is 2026 AI slop. ironically, in part introduced by anthrophic's frontend-design skill, which replaced one set of model defaults...with another (fraunces for editorial anyone? warm brown italics?) it's incredibly hard to avoid, and build software and skills that unlock true creativity.
>
> — Paul Bakaus (@pbakaus), [May 3, 2026](https://x.com/pbakaus/status/2050766371081081133)

![](https://pbs.twimg.com/media/HHXG_fBaYAADkgF?format=jpg&name=large)

> "漂亮"这件事的标准其实不断在被重写，当人们都到平均水平时，原来的漂亮就不再是漂亮。

审美重要吗？当然重要。新 UI 真的看起来更冷淡，但是大家都不喜欢。

今天，无论是 Claude Code 还是 Cursor / Codex 都支持模型读取 Preview 的能力。可以预见的将来，模型能力会有更好的视觉，毕竟偏科生 Gemini 已经展现了不错的视觉理解能力。

软件不是给 AI 设计的，Graphical user interface 始终是给人用的。设计从来是"以人为本"，而不是以 LLM/Agent 或者 AI 为本。

毕竟付钱的始终是你我这样的人。
